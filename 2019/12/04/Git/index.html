Git

版本控制工具应该具备的功能
  • 协同修改
    • 多人并行修改服务器的同一个文件
  • 数据备份
  • 版本管理
    • SVN ：增量式管理
      • 每次迭代，保存变动的部分，读取版本的时候（拼接原始版本和变动部分）
    • GIT: 文件系统快照
      • 如果没有修改，则引用原文件（创建新的指针，指向同一个内存）
  • 权限控制
  • 历史记录
  • ⭐分支管理
版本控制工具 - 集中式 vs 分布式

集中式 ：版本库存在远程的中央服务器上，干活的时候，从远程服务器下载文件到自己电脑上，干完了，提交文件到远程服务器。(要是断网了都没法工作了🤦)。要是中央服务器挂了，那就尴尬了。（单点故障）

  • 比如SVN

分布式：分布式版本控制系统根本没有“中央服务器”，每个人的电脑上都是一个完整的版本库。这样，就不需要联网了。 但是两个之间如何协作呢？分布式版本控制系统通常也有一台充当“中央服务器”的电脑，但这个服务器的作用仅仅是用来方便“交换”大家的修改 。

⭐Git基本操作

  • git init ： 初始化仓库，会生成隐藏文件 .git (存放本地库相关的文件)

  • git config 设置签名

    • 形式 ： 用户名 + Email地址

    • 主要是用来区分不同的开发人员（总得知道是谁push的代码）

    • 局部签名和全局签名

      • 项目级别/仓库级别：仅在当前本地库范围内有效

      输入配置命令后，信息保存在当前本地库的 ./.git/config中

      • 系统用户级别：登录当前操作系统的用户范围 （window xxx用户）

      输入命令后，信息保存在C盘的 ~/.gitconfig 文件

      局部签名优先级高于系统级别，操作远程库时，先在 ./.git/config寻找签名，如果没有，

      就在在C盘的/.gitconfig中找，将找到签名作为标识符，用来辨别是哪个开发人员（二选一，一定要设置签名）

  • git status ： 查看工作区，暂存区状态

  • git add [file name] ： 添加 “新建/修改” 到暂存区

  • git commit -m “commit message” [filename] ： 将暂存区内容提交到本地库

  • git log ：查看历史记录

    • 历史纪录是一张哈希表，每个记录对应一个索引

    • 其他格式显示历史纪录 ：

      • gitlog–pretty=oneline
      • gitlog–oneline
      • gitreflog

  • 切换历史纪录（本质上是移动指针，指向不同的索引对应的记录）

    切换方式

    • 基于索引值操作

      • git reset –hard [局部索引值]
      • git reset –hard a6ace91

    • 使用^符号：只能后退

      • git reset –hard HEAD^
      • 注：一个^表示后退一步，n 个表示后退 n 步

    • 使用~符号：只能后退

      • git reset –hard HEAD~n
      • 注：表示后退 n 步

    • reset 参数对比 （是否同步指针）

      git根据三个区指针的指向第几个版本，来判断进度，比如本地库和暂存区后退一个版本（指针向后移了一位），相当于工作区版本领先了一个版本，需要git add ，git commit，同步暂存区和本地库的版本为最新版

      • –soft
        • 仅仅在本地库移动 HEAD 指针
      • –mixed
        • 在本地库移动 HEAD 指针
        • 重置暂存区
      • –hard
        • 移动本地库的HEAD指针
        • 重置暂存区
        • 重置工作区

  • 删除文件并找回 (实际上就是移动指针，指针移动到删除前的某个历史版本)

    • 前提：删除前，文件存在时的状态提交到了本地库（commit）。 （总得在hash表中生成对应的历史记录，才能通过索引查找到记录啊）
    • 操作：git reset–hard [指针位置]
      • 删除操作已经提交到本地库：指针位置指向历史记录
      • 删除操作尚未提交到本地库：指针位置使用 HEAD

  • git diff : 比较文件差异（比较当前工作区文件和暂存区或者历史纪录）

    • git diff [文件名]
      • 比较工作区和暂存区的
    • git diff [本地库历史版本] [文件名]
      • 比较工作区和本地库历史纪录
分支管理

git可以创建分支，在不同的分支上干活，多条线同时推进多个任务

比如A在web_nav分支上写导航栏代码，B在web_footer分支上写页脚，写完了，就将分支合并到master分支。

如果两个分支上都同时修改了同一行代码，合并的过程中就会产生冲突，git会在冲突文件中标记冲突的位置。那如何解决冲突？自己手动解决呗😂（机器也不知道选哪段代码…）

![branch manege](branch manege.png)

分支操作

  • 创建分支

    git branch [branch name]

  • 查看分支

    git branch -v

  • 切换分支

    git checkout [branch name]

  • 合并分支

    • git checkout 主干道上（需要接收修改的分支上）

    • git merge [branch name]

      比如： git checkout master , git merge web_nav

  • 解决冲突

    • 冲突报错，冲突文件中标记了冲突位置

  • 冲突的解决

    • 第一步：编辑文件，删除特殊符号
    • 第二步：把文件修改到满意的程度，保存退出
    • 第三步：gitadd[文件名]
    • 第四步：gitcommit-m”日志信息”
      • 注意：此时 commit 一定不能带具体文件名

  • 本地创建分支，上传分支到远程仓库

    • git branch [branchName] 创建本地分支

    • git checkout [branchName] 切换到分支

      • 在分支上添加代码，在分支完成指定代码

    • git push origin [branchName] 将分支提交到指定的远程仓库

      • origin (远程仓库地址的别名 ) ： git remote add aliasName url

    • git pull origin [branchName] ， git checkout branchName （拉取仓库，切换到分支）

Git基本原理

哈希

哈希是一个系列的加密算法，各个不同的哈希算法虽然加密强度不同，但是有以下 几个共同点：

①不管输入数据的数据量有多大，输入同一个哈希算法，得到的加密结果长度固定。

②哈希算法确定，输入数据确定，输出数据能够保证不变

③哈希算法确定，输入数据有变化，输出数据一定有变化，而且通常变化很大

④哈希算法不可逆 Git 底层采用的是 SHA-1 算法。

哈希校验文件

在远程和本地使用同一个hash算法，对同一个内容进行hash计算，如果得到的结果相同，说明传输的文件时完整的，没有受损（hash网络传输校验）。

Git就是靠这种机制来从根本上保证数据的完整性

![hash check](hash check.png)

Git保存版本机制

Git 把数据看作是小型文件系统的一组快照。每次提交更新时 Git 都会对当前 的全部文件制作一个快照并保存这个快照的索引。为了高效，如果文件没有修改， Git 不再重新存储该文件，而是只保留一个链接指向之前存储的文件。所以 Git 的 工作方式可以称之为快照流。

github

在github上登陆注册帐号，创建远程仓库

本地仓库关联远程仓库地址

  • 创建远程库地址别名

    • git remote -v 查看当前所有远程地址别名
    • git remote add [别名] [远程地址]
    • git remote add orign url

  • 推送

    • git push [别名] [分支名]
    • git push origin master
      • 第一次push，需要指定push到哪个分支

  • 克隆

    • git clone [远程地址]
    • 把完整的远程库下载到本地
    • 创建origin远程地址别名
    • 初始化本地库

  • 拉取

    • pull = fetch + merge
    • git fetch[远程库地址别名] [远程分支名] ： 将远程库下载到本地
    • git merge [远程库地址别名/远程分支名] ： 和本地库合并，可能会产生冲突
    • git pull [远程库地址别名] [远程分支名]

  • 解决冲突

    • 如果不是基于Github远程库的最新版本所作的修改，不能推送，必须先拉取
    • 拉取下来，如果进入冲突状态，按照分支冲突解决解决
      • git push ： 命令行报冲突错误
      • git pull，然后手动解决冲突
      • 解决完之后，git add ，git commit

github设置SSH

  1. cd ~ ： 进入当前用户目录

  2. rm -rvf .ssh ： 删除.ssh目录

  3. ssh-keygen -t rsa -C xxx邮箱（注册github的邮箱） ： 生成公私钥对

  4. cd .ssh

    ls -lF

  5. cat id_rsa.pub 查看公钥内容

  6. 复制 id_rsa.pub 文件内容，登录 GitHub，点击用户头像→Settings→SSHandGPG keys

  7. New SSH Key，输入复制的公钥信息

  8. 然后git push ，就不用输入密码了
